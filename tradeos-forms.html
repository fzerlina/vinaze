Every form field is composed of the same four layers. All four are always present structurally — even if some are visually hidden when unused.

①Label — always above the input, never inside (placeholder is not a label). Font: Manrope 11px, weight 600, color #555. Required asterisk in red, optional tag in muted.
②Input — 38px height, 6px border-radius, border #d8d6d0. Font: Manrope 13px for text, Inconsolata 13px for numbers. See §02 for states.
③Hint — appears below the input, always. Font: Manrope 11px, color #bbb. Provides format guidance or constraints. Never repeats the label.
④Error message — replaces or supplements the hint on validation failure. Red #e0400a, 11px, prefixed with an error icon. Never shown before the user has interacted.
Label conventions
·Always sentence case — "Nama vendor", not "Nama Vendor" or "NAMA VENDOR". Exception: proper nouns and acronyms (NPWP, PPN).
·Required fields use a red asterisk * after the label. Required is the default for most ERP fields — mark optional instead where relevant.
·Never use placeholder as a substitute for a label. Placeholders disappear on input and break accessibility. Use them for format examples only: e.g. 01.234.567.8-091
·Label spacing: 5px gap between label and input. 4px gap between input and hint/error. 18px gap between field groups.
·In-table edit mode: column header acts as the label — no separate label per cell. Validation error appears as a tooltip on the cell border.

05Validation
Validation
When to validate, what to show, and how to write error messages — rules that apply across all field types.
Timing rules
·Validate on blur — not on every keystroke. Showing errors while the user is typing is hostile.
·Validate on submit — catch any fields the user skipped. Scroll to and focus the first error.
·Clear the error on next focus — give the user a clean slate when they return to fix a field.
·Never validate on mount — showing errors on a freshly opened form is confusing.
·Real-time validation is only acceptable for fields with a fixed format where the user benefits from immediate feedback: NPWP (15 digits), NSFP (specific pattern).
Error message writing
·Say what's wrong and how to fix it. "Format tidak valid" alone is not enough. "Format NPWP tidak valid — gunakan format 00.000.000.0-000.000" is correct.
·Never blame the user. "Anda memasukkan nilai yang salah" → "Nilai harus berupa angka positif".
·Sentence case, no period at the end of short error messages.
·One error per field — show the most important failure, not a list. Fix order: required → format → business rule.
·For server-side errors (e.g. duplicate NPWP), show a distinct style — same red border, but the message includes a contextual link: "NPWP ini sudah terdaftar — lihat vendor yang ada".

06Form Layout
Form Layout
How fields are arranged, grouped, and submitted. ERP forms are dense by nature — structure prevents them becoming overwhelming.
Grid rule: Forms use a 2-column grid as default. 3-column for short, related fields (day / month / year, DPP / PPN / total). Never more than 3 columns — the inputs become too narrow for financial data. Single-column for long text, keterangan, or address blocks.

Action rules
①
Primary action always leftmostThe submit/save button is always at the left of the action row. Destructive actions (delete, void) are never in the action row — they live in a separate danger zone or are accessed via a menu.
②
Cancel is a text link, not a ghost buttonCancel has no consequence — it doesn't deserve the visual weight of a button. Text link with indigo color, left-aligned within the actions row, after the secondary button.
③
Calculated fields are read-only, visually distinctPPN, total, running balance — any field computed from other inputs uses the readonly state (#faf9f7 bg). Never disable these fields; the user should still be able to select and copy the value.
④
Section labels group related fieldsUse a small uppercase label (10px, #bbb) above a logical group. Avoid full section dividers for short forms — they create excessive visual hierarchy on simple entry screens.

08ERP-Specific Patterns
ERP-Specific Patterns
Field patterns that are unique to Indonesian ERP and tax contexts. These are not generic form patterns — they encode domain rules.
NPWP field
15 digits · fixed format
NPWP
Format: 00.000.000.0-000.000 (15 digit)
NPWP
NPWP harus 15 digit — format 00.000.000.0-000.000
·Auto-format on input: insert dots and hyphen as the user types. Never require the user to type the separators manually.
·Validate against DJP regex pattern on blur. Show real-time character count: 9/15
·Inconsolata font, letter-spacing 0.04em for readability of the formatted string.
NSFP field
Nomor Seri Faktur Pajak
NSFP optional
Format DJP: 000.000-YY.XXXXXXXX — hanya angka, diisi otomatis saat upload e-Faktur
·NSFP is a string, not a number — never apply number formatting or arithmetic.
·Optional on entry — system fills it from e-Faktur upload. Only manually entered if the user has the number from a printed faktur.
·Validate against DJP regex: ^\d{3}\.\d{3}-\d{2}\.\d{8}$

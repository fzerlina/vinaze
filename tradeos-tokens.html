Format rules
·Arrow icon always leads ▲ for up, ▼ for down. Never use + or – alone as the direction signal. Arrow always left of the number.
·Percentage in Inconsolata, comma decimal — +5,2% not +5.2%. Matches the IDR number format convention.
·Comparison label in Manrope, smaller — "vs bln lalu", "vs anggaran", "YTD". Always present. A percentage without a baseline is meaningless.
·No background — color is carried by the text and arrow only. No pill, no chip background. Keeps delta indicators lightweight and unobtrusive beside figures.
·Absolute + relative when space allows — e.g. +Rp 23 jt · +5,2%. Absolute first, percentage second.
Direction-awareness — the critical rule
!Color encodes favorability, not sign. A cost increase (+) is red. A revenue increase (+) is green. The same ▲ can be green or red depending on what is being measured.
!Never auto-apply green = up, red = down. Expenses, DSO, overdue count — an increase is bad. The component must know the metric’s direction.
·When directionality is ambiguous or unknown — use neutral muted (#888, no background). Do not guess.
·Zero delta: show 0,0% in neutral, no arrow. Nothing changed — no color, no icon.

✓ Do
  • Always include a baseline label — "vs bln lalu", "vs anggaran", "YTD"
  • Text color only — no background, no border. Arrow + Inconsolata figure in green or red.
  • Show absolute + percentage when space allows: +Rp 23 jt · +5,2%
  • Apply direction-awareness: cost up = red, revenue up = green
  • Inline chips in AI narrative use the same token colors as standalone chips
  • Zero delta: 0,0% in neutral, no arrow
✗ Don’t
  • Auto-color every upward number green — an expense increase is unfavorable
  • Show a percentage without its baseline — no context, no meaning
  • Add a pill background to delta indicators — use text color only
  • Use + and – signs alone as the direction signal without an arrow
  • Apply delta chips inside GL rows or transactional tables
  • Show a chip when comparison period has no data — use neutral dash instead

Key rule: Number colors (green/red/amber) encode direction — favorable, unfavorable, at-risk. They are never used on debit or credit entries in GL views, where column position already carries that meaning. Blue is reserved for running balances on non-GL dashboards. Status colors are for state pills only — never mix the two families.
How major ERPs handle color in financial tables
Reference — color usage by system & context
GL · P&L · Dashboard
System GL — Dr & Cr Negative values Dashboard KPIs Source
Xero Plain black — no color on Dr/Cr columns. Column label carries the meaning. Red for amounts less than zero in reports. Configurable. Green/red arrows for favorable/unfavorable variance vs prior period. Xero GL UI
QuickBooks Online Plain black — no Dr/Cr color. Separate Debit and Credit columns, amounts in black. Red for negative amounts — user-selectable in report settings alongside parentheses and minus sign. Green for income, red for expenses on home dashboard charts. Not in transactional views. QBO Report Settings
NetSuite Plain black in GL Detail. No color distinction between Dr and Cr entries. Parentheses as default for negative values in financial statements (GAAP convention). Red optional. Color-coded KPI tiles: green for on-target, amber for warning, red for critical. Not applied to GL rows. NetSuite GL UI
SAP S/4HANA Plain black in FI line items. Dr/Cr indicated by posting key and column header only. Parentheses in financial reports (IFRS/GAAP). Red highlighting configurable at report level. Fiori dashboards use semantic color: green/amber/red for KPI status. Strictly separated from transactional tables. SAP Fiori Design
Odoo 17 Plain black for Dr/Cr in journal entries. No color on transaction amounts. Red for negative account balances in balance sheet views. Green/red for profit/loss indicators on dashboard tiles. Odoo Accounting UI
The consistent pattern across all five: GL and journal entry views use plain black for both Dr and Cr — no color. Color appears only at the reporting and dashboard layer, where it encodes direction of variance (favorable vs unfavorable) or severity (on-target / warning / critical). The separation is not stylistic — it prevents accountants from misreading structural accounting entries as directional signals.

✓ Do
  • Dot → compact list rows where space is tight
  • Pill → document-level state (invoice, bill header)
  • Text status → field values that carry inherent meaning
  • Tax text → tax codes and rates only, never other metadata
✗ Don't
  • Use pills for every type of metadata — they lose meaning fast
  • Stack multiple pills or dots in one row
  • Use number colors (blue/green/red) for status pills
  • Apply tax text style to non-tax annotations

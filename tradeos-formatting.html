01Principles
Principles
Four rules that govern every formatting decision. When in doubt, come back to these.
01
Clarity over precision
Show only as many decimals as the context demands. P&L views suppress trailing zeros. Transaction views always show two decimals. Never show more than two decimal places in user-facing IDR.
02
Consistency within a view
Every number in the same table follows the same rules — same decimal count, same negative convention, same alignment. Never mix formats within a single table or screen.
03
Context-driven format
Tabular views and KPI cards have different rules. A number that belongs in a column header belongs nowhere else in that column. Match the format to the reading context, not to a single universal rule.
04
Indonesian-first
IDR uses period as thousands separator and comma as decimal separator. This is the inverse of USD/USD-influenced software defaults. Never apply US separators to IDR figures.
02Default Format
IDR — Default Format
The canonical format for all Rupiah figures. Context determines whether the symbol appears inline or only in the column header.
Canonical formatIDR
Rp 1.250.000,00
·Symbol: Rp followed by a single space, then the digits
·Thousands separator: period . — e.g. 1.250.000
·Decimal separator: comma , — e.g. 1.250.000,00
·Decimal places: always 2 in transactional views; suppressed in P&L unless non-zero cents exist
·In tables: Rp lives in the column header only — never repeated per cell
·Outside tables (cards, narrative): Rp inline as prefix — Rp 1.250.000,00
✓ Do
Rp 1.250.000,00
Inline / KPI card context. Symbol + space + figure + 2 decimals.
✗ Don't
Rp1,250,000.00
US separators applied to IDR. Comma is not an IDR thousands separator. Period is not an IDR decimal separator.
✗ Don't
IDR 1.250.000
ISO code in UI. Use Rp in display; IDR is for APIs and data exports only.
✗ Don't
Rp1.250.000
Missing space. The standard is Rp (symbol + space) when used inline, per Indonesian convention.
In tables: the Rp symbol belongs in the column header, not repeated on every cell. This reduces visual noise and follows ERP convention (SAP, Oracle, Odoo all do this).

03Negative Values
Negative Values
PSAK and Indonesian accounting practice use parentheses for negative amounts — not the minus sign. The choice of convention must be consistent within any single view.
Convention by context
·Financial statements & P&L → parentheses: (1.250.000,00) — PSAK convention
·Transactional views (GL, journals, AR/AP) → parentheses preferred; minus sign acceptable if the system can't render parentheses
·KPI cards & dashboards → minus sign -Rp 1.250.000,00 with red color (#e0400a) — more immediately readable at a glance
!Never mix parentheses and minus signs within the same table — choose one and apply it to all rows
·Negative values are always rendered in red (#e0400a) regardless of convention used
·In column headers, do not prefix the column with a minus — the cell value carries the sign
✓ Financial statements
(1.250.000,00)
Parentheses, no Rp in cell (already in header). Red color.
✓ Dashboard / KPI card
-Rp 1.250.000,00
Minus sign with inline Rp. Red color. Easier to parse at a glance.
✗ Don't
-1.250.000,00
Minus without color. In a dense table, a minus sign without red color is easy to miss.
✗ Don't mix in same table
(500.000,00)   -800.000,00
Inconsistent convention. Two different negative formats in the same table breaks scannability.

04Zero & Missing
Zero & Missing Values
Two separate concepts — zero is a real value; missing means no data exists. They must never look the same.
Zero values — by context
·P&L and financial statements: display as - (en dash, muted color) — suppresses visual noise in summary views
·Transactional views (GL, AP, AR, invoices): display as 0,00 — keeps decimal alignment consistent across all rows
·KPI cards: display as Rp 0 — never Rp 0,00. Zero is zero; decimals add no information here.
·Zero values are rendered in muted color (#bbb) in all contexts
Missing / unavailable values
·Use em dash — for values that are genuinely not applicable or not yet available
·Never use a hyphen - for missing — it is visually identical to a minus sign in a table
·During loading: show a skeleton bar, never Rp 0 or —
·If data fetch fails: show — with an error tooltip — not blank, not Rp 0
·Em dash is always right-aligned and muted in a numeric column — it occupies the same column space as a number

05Table Rules
Table Alignment & Totals
Column alignment and total row styling are the two most commonly broken rules in ERP interfaces. Both are non-negotiable for financial readability.
Alignment rules
·All numeric columns are right-aligned — without exception. Left-aligning amounts breaks decimal scanning.
·Decimal alignment must be vertically consistent — all rows in a column must show the same number of decimal places so commas stack in a vertical line
·Text / description columns are left-aligned — account names, vendor names, reference numbers
·Use Inconsolata (monospace) for all numeric cells — ensures equal character width and perfect decimal stacking
·Column widths should be wide enough to never truncate a figure — figures that wrap or truncate are an audit risk
Total row styling
·Subtotals: bold text + single top border (border-top: 1px solid). Slightly tinted background (#f5f4f1).
·Grand totals: bold text + double bottom border (border-bottom: 3px double) + single top border. Darker tinted background (#edeae3). This is the PSAK / accounting convention for final totals.
·Never use background color alone to indicate a total — always pair with the border convention
·Section headers (Pendapatan, Beban, etc.) get a light separator row — not a total row style
·Indent sub-items 16px under their section header to create visual grouping without extra dividers

06PPN / DPP
PPN & DPP Display
All tax-related tables must show DPP, PPN, and Total as separate columns — never collapsed into a single amount. This is required for e-Faktur compliance and DJP audit readiness.
Required columns
·DPP (Rp) — Dasar Pengenaan Pajak, the taxable base. Always shown separately.
·PPN 11% (Rp) — the VAT amount. Label must include the rate (11%) explicitly so users can verify the calculation.
·Total (Rp) — DPP + PPN. This is the final invoice/payable amount.
·For PPh withholding: add a PPh (Rp) column showing the withheld amount in parentheses, and a Net Bayar (Rp) column for the actual cash outflow
·NSFP (Nomor Seri Faktur Pajak) should appear as a separate non-numeric column — left-aligned, Inconsolata font
·PKP status of the vendor determines whether PPN applies — show — in the PPN column for non-PKP vendors
PPh withholding column pattern
·PPh 23 withholding is shown as a deduction in parentheses — e.g. (200.000,00) — red color
·Column header: PPh 23/2% (Rp) — include the rate in the header
·Net Bayar = Total − PPh withheld. This is the actual bank transfer amount.
·PPN and PPh are separate tax obligations — never combine them into a single "Tax" column

07Multi-Currency
Multi-Currency Tables
Foreign currency transactions must always show three columns: the original FCY amount, the BI Middle Rate (Kurs), and the IDR equivalent. Never collapse these into one.
Required columns — FCY transactions
·FCY Amount: native currency amount with its own format rules — e.g. $12,500.00 for USD, €10,000.00 for EUR
·Kurs (Rp): BI Middle Rate used for the conversion — 15.750,00. Date of rate must be disclosed in the table header or footnote.
·IDR Equivalent (Rp): FCY × Kurs, right-aligned, full IDR format with 2 decimals
·Currency column header must include the ISO code: Nilai (USD), Kurs BI (Rp/USD), IDR Equiv. (Rp)
·If a table mixes multiple foreign currencies, add a Currency column (text, left-aligned) before the FCY amount
Currency format by locale
·USD: $12,500.00 — comma thousands, period decimal, prefix symbol, no space
·EUR: 10.000,00 € — period thousands, comma decimal, suffix symbol with space. European convention.
·SGD: S$12,500.00 — use S$ to distinguish from USD
·Never apply IDR separators to foreign currencies — each currency follows its own locale rules
·Kurs (exchange rate) always uses IDR format — it is expressed as Rupiah per 1 unit of FCY

08Display Modes
Display Modes
Context determines how much of the number to show. ERP table views use full format. Dashboards and KPI cards use compact or rounded.
Mode comparison
Mode Format Use when Never when
Standard 1.250.000,00 All ERP tables, GL, AP/AR, invoices, financial statements KPI cards, chart axis labels, mobile compact lists
Compact Rp 1,2 jt  /  Rp 3,4 M Dashboard KPI cards, summary widgets, chart labels, mobile Any table where precision matters — invoices, GL, tax views
Rounded 1.250.000 When source has cents but context doesn't need them. Round, never truncate. Exchange rates, tax amounts, any value where cents are contractual
High Precision 15.750,2500 Exchange rates (up to 4dp), unit prices in trade docs, API/export contexts Any user-facing dashboard, summary card, or compact display
⚠ Indonesian suffix: M = Miliar (10⁹), not Million (10⁶). In Indonesian, jt = juta (million), M = miliar (billion). Do not use M where English speakers would read it as million — this causes a 1000× error. Use jt for millions, M for billions.

09Interaction Rules
Interaction Rules
Input fields
·User types raw digits — no separators during typing
·Format on blur — apply separators and symbol when field loses focus
·On focus — strip formatting, show raw digits for clean editing
·Placeholder: e.g. 1.250.000 — never Rp 0,00
·Accept both separator styles on input; normalize to IDR on blur
Loading states
·Never show Rp 0 during loading — implies real zero balance
·Use a skeleton bar — 8–10 chars wide for standard IDR
·Never show — during loading — em dash means N/A, not pending
·If data fetch fails: — with error tooltip
Live updates
·Debounce at 300–500ms minimum for polling-updated figures
·Highlight briefly on change — fade from tint to neutral
·Direction indicator ↑ ↓ fades after 2s — never permanent
·Invoice builder running totals: update immediately, no debounce
10Component Spec
Component Spec — Currency
A single Currency component centralises all formatting logic. Pass raw numbers and intent — never pre-formatted strings.
Prop Type · Default Values / Notes
valuenumber · —Raw numeric value. Never pass a pre-formatted string. Pass 1250000, not "Rp 1.250.000,00".
currencystring · 'IDR'ISO 4217 code. Controls separator style, symbol, and symbol position. IDR USD EUR SGD
formatstring · 'standard'standard compact rounded precision — see §08 for when to use each.
decimalsnumber|'auto' · 'auto'auto = show only when non-zero cents exist. 0 = no decimals. 2 = always 2dp. 4 = high precision.
showSymbolboolean · trueSet false when currency is already shown in the column header. Prevents symbol repetition in every table cell.
negativestring · 'parentheses'parentheses = (1.250.000,00) for financial statements. minus = -Rp 1.250.000,00 for dashboards.
zerostring · 'value'value = show 0,00. dash = show - (for P&L views). Never use dash in transactional tables.
loadingboolean · falseWhen true, render a skeleton bar. Never render Rp 0 or — during loading.
emptystring · '—'Rendered when value is null or undefined. Default em dash. Never use Rp 0 as a null fallback.
11Gaps & Additions
Gaps & Additional Rules to Consider
Rules not yet covered in this document that are commonly needed in ERP-context Indonesian finance interfaces.
⚠ Gap — Rounding method
The document does not yet define which rounding method to use when converting FCY to IDR or when rounding tax amounts. PSAK and DJP both require specific rounding — typically half-up (commercial rounding). Confirm with your tax/accounting team whether half-up, half-even (banker's rounding), or floor rounding applies to PPN calculations. DJP often rounds PPN to the nearest whole Rupiah.
⚠ Gap — Date format for exchange rates
Multi-currency tables use BI Middle Rate, but which date's rate must be disclosed. Per PSAK 10, the transaction date rate is used for initial recognition; the period-end rate for balance sheet translation. Add a footnote or header annotation to every multi-currency table showing: Kurs BI Middle Rate per [DD MMM YYYY].
⚠ Gap — Thousands display for amounts below 1.000
The document assumes values ≥ 1.000. For values below 1.000 (e.g. 850,00), no thousands separator is needed. Confirm this is applied correctly — some systems erroneously show 0.850,00. The correct form is simply 850,00.
⚠ Gap — Print / PDF export formatting
Screen formatting rules don't always carry through to PDF exports. Define explicitly whether PDF invoices and reports follow the same rules, including the double-underline for grand totals, monospace font for numbers, and symbol-in-header convention. PDF/print often defaults to system fonts that break Inconsolata alignment.
⚠ Gap — Sorting behavior for formatted columns
If users can sort columns containing formatted IDR values, sorting must operate on the raw numeric value, not the formatted string. Rp 1.250.000,00 sorted lexicographically will produce wrong results. Ensure the data layer passes raw numbers and formatting is display-only.
⚠ Gap — Accessibility for negative values
Parentheses alone are not accessible — screen readers may not announce them as negative. For accessibility compliance, pair parentheses with a red color AND an aria-label attribute explicitly stating the value is negative: aria-label="negative Rp 1.250.000,00". Color alone is never sufficient per WCAG 1.4.1.
⚠ Gap — Tax invoice number (NSFP) formatting
NSFP (Nomor Seri Faktur Pajak) has a DJP-defined format: 000.000-YY.XXXXXXXX (e.g. 010.000-25.00000041). This is a string, not a number — never apply number formatting to NSFP. Display left-aligned in Inconsolata. Validate against the DJP regex on input.
